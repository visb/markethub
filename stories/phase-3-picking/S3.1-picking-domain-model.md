# S3.1 Modelo de domínio de picking
- **Fase:** 3
- **Epic:** Modelo de separação
- **Status:** done
- **Depende de:** [S2.1, S2.7]

## Objetivo
Modelar separação: tarefa por sub-pedido, item separado, substituições e caixas.

## User story
Como sistema, preciso de entidades de separação para rastrear cada item desde o pedido pago até a entrega ao entregador.

## Critérios de aceite
- `PickTask` (1:1 com `OrderGroup`) com `pickerId` (`StoreStaff`/`User`), `status`, timestamps
- `PickItem` (1:1 com `OrderItem`) com `status`: `pending | picked | refused | substituted`
- `quantityPicked` / `weightGramsPicked` (suporta peso em g para `saleType=weight`)
- `Substitution`: `OrderItem` original → `Offer` substituto, snapshot de preço, diferença, `approvalStatus`
- `Box` (caixa) com `serial`, `passcode` (senha numérica) e payload de QR, vínculo ao `PickTask`, lista de `PickItem` (ref: `Receive.jpg`)
- Enum `PickTaskStatus`: `queued → assigned → picking → packed → ready_for_pickup`
- Estender `OrderStatus` com `ready_for_pickup` (entre `picking` e `on_the_way`)
- Index: `PickTask` por loja(`storeId`)+status, por `pickerId`

## Escopo / Fora de escopo
- **Fora:** lógica de atribuição (S3.2) e UI (S3.7)
- **Inclui:** entidades, relações, enums, índices, migration Prisma; tipos em `packages/types`

## Notas técnicas
- `PickTask` deriva de `OrderGroup`; `OrderGroup.status` reflete progresso (picking → ready_for_pickup)
- Snapshot de preço na `Substitution` (não referência viva à `Offer`)
- `Box.serial` único; QR codifica `boxId`; `passcode` exibido ao cliente p/ confirmação na entrega (ref: `Receive.jpg`)
