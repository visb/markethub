# S2.6 — Pagamento PIX

- **Fase:** 2
- **Epic:** Marketplace (cliente)
- **Status:** todo
- **Depende de:** [S2.5]

## Objetivo
Cobrar o pedido via PIX: QR dinâmico, expiração, confirmação por webhook e atualização de status.

## User story
Como cliente, quero pagar por PIX e ver a confirmação, para concluir o pedido (ref: `Payment.jpg`, `Confirmed.jpg`).

## Critérios de aceite
- [ ] `PaymentProvider` (interface) + provider concreto do gateway escolhido + `MockPaymentProvider` (dev/test).
- [ ] `POST /orders/:id/pay` cria cobrança PIX → retorna QR Code + copia-e-cola + expiração (ex.: 30 min).
- [ ] `Payment` persistido com status (`pending`/`paid`/`expired`/`failed`) e ids do gateway.
- [ ] Webhook `POST /webhooks/pix` valida assinatura, marca pagamento `paid` e pedido `paid` (idempotente).
- [ ] `GET /orders/:id/payment` para polling de status ("Aguardando pagamento…" → "Pagamento realizado").
- [ ] Pagamento confirmado dispara o fluxo de pedido (S2.7): status `preparing` + push ao ERP.

## Escopo / Fora de escopo
Inclui: criação de cobrança PIX, webhook, status.
Não inclui: estorno/cancelamento avançado, split de pagamento (backlog).

## Notas técnicas
- **Pendência:** confirmar gateway PIX (Mercado Pago / Asaas / Efí). Provider isolado permite trocar.
- Segredos via env (`PIX_PROVIDER`, `PIX_API_KEY`); webhook com verificação de assinatura.
- Idempotência forte no webhook (gateway reenvia). Reusa formato de erro/log da Fase 0.
