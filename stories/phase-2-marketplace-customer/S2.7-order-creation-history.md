# S2.7 — Pedido: criação, status e histórico

- **Fase:** 2
- **Epic:** Marketplace (cliente)
- **Status:** todo
- **Depende de:** [S2.6, S1.2]

## Objetivo
Ciclo de vida do pedido após pagamento: empurrar ao ERP, expor status e histórico ao cliente.

## User story
Como cliente, quero acompanhar e rever minhas compras, para saber o status e o histórico (ref: `Orders.jpg`, `Order Created.jpg`).

## Critérios de aceite
- [ ] Pagamento confirmado → `Order` vira `preparing`; cada `OrderGroup` é empurrado ao ERP do merchant via `ErpConnector.pushOrder`.
- [ ] `GET /orders` (cliente): lista paginada com status, total, endereço, data ("MINHAS COMPRAS").
- [ ] `GET /orders/:id`: detalhe com itens por merchant, totais, status, janela de entrega.
- [ ] Estados refletem o pipeline: created → paid → preparing → picking → on_the_way → delivered (Fases 3/4 avançam).
- [ ] Transições de status validadas (máquina de estados); auditáveis.
- [ ] Cancelamento básico antes de `preparing`.

## Escopo / Fora de escopo
Inclui: pós-pagamento, push ERP, listagem/detalhe, máquina de estados.
Não inclui: rastreio em tempo real (Fase 5), separação/entrega (Fases 3/4).

## Notas técnicas
- Reusa `ErpConnector.pushOrder` (stub hoje no CSV) + auditoria de envio.
- Avanço de status por picking/delivery vem nas próximas fases; aqui só created→preparing.
- Idempotência no push (não reenviar o mesmo grupo).
