# S5.7 Navegação admin Mercados → Lojas → filhos
- **Fase:** 5
- **Epic:** Admin
- **Status:** done
- **Depende de:** [S3.9, S3.10, S3.11, S5.4]

## Objetivo
Dar ao admin uma visão hierárquica do negócio: lista de **mercados (merchants)** → **lojas** de cada mercado → dentro da loja, **produtos/ofertas, pedidos, funcionários e dados/operação**. Antes o admin só tinha listas globais planas (Pedidos, Catálogo, Usuários) sem drill-down pela estrutura.

## User story
Como admin, quero navegar mercado → loja → (produtos, pedidos, funcionários, dados), para inspecionar e ajustar cada loja sem precisar ser staff dela.

## Critérios de aceite
- **Mercados:** `GET /admin/merchants` (com `?search=`) lista nome, slug, #lojas, taxas; `GET /admin/merchants/:id` traz o mercado + lojas com #ofertas/#staff. CRUD: `POST /admin/merchants` (slug auto do nome, conflito 409), `PATCH /admin/merchants/:id` (nome, slug, taxas, ativo).
- **Loja:** `GET /admin/stores/:id` traz endereço, geo, mercado e counts (ofertas, staff, slots, pedidos por status). CRUD: `POST /admin/stores`, `PATCH /admin/stores/:id` (nome, endereço completo, lat/long, externalId), `PATCH /admin/stores/:id/active`.
- **Produtos/ofertas:** `GET /admin/stores/:id/offers` (paginado, `?search/page/pageSize`, ofertas + estoque por loja, **sem** escopo de manager); edição via `PATCH /admin/stores/offers/:offerId` e `/admin/stores/stocks/:stockId`, mantendo `lockedFields`; destravar via `DELETE .../locks/:field` (devolve o campo ao sync ERP).
- **Pedidos:** aba reusa `GET /admin/dashboard/orders?storeId=` (paginado) + drawer de detalhe via `GET /admin/dashboard/orders/:id`.
- **Funcionários:** `GET /admin/stores/:id/staff`; criar (`POST /admin/users` com `storeId`), ativar/desativar vínculo (`PATCH /admin/stores/staff/:staffId/active`), remover vínculo (`DELETE /admin/stores/staff/:staffId`); busca client-side.
- **Slots de entrega:** gestão na aba Dados — `GET /store/slots?storeId=`, `POST /store/slots`, `DELETE /store/slots/:id` (bloqueia se houver reserva).
- **Dados/Operação:** edição da loja + slots + métricas de `GET /admin/dashboard/operations?storeId=`.
- Frontend Vite: nav "Mercados" + rotas `merchants`, `merchants/:merchantId`, `stores/:storeId`; `StoreDetail` com abas Produtos/Pedidos/Funcionários/Dados e breadcrumb.

## Escopo / Fora de escopo
- **Inclui:** endpoints admin de merchant/loja, CRUD de mercado e loja, edição de oferta/estoque (com unlock), gestão de funcionários (criar/ativar/remover vínculo), gestão de slots de entrega, detalhe de pedido, telas de drill-down com busca/paginação.
- **Fora:** exclusão de mercado/loja (só ativar/desativar); edição de produto canônico (fica no Catálogo, S1.7); reatribuição de staff entre lojas.

## Notas técnicas
- Stack admin é **Vite + React SPA** — mantido.
- `AdminMerchantsService` é variante **sem escopo de StoreStaff** (admin enxerga qualquer loja), diferente de `MerchantService` (escopado ao manager logado). A lógica de `lockedFields` de oferta/estoque foi espelhada.
- Os dois controllers em `admin/stores` (index em `users.module` + detalhe em `admin.module`) convivem por terem rotas exatas distintas.
- `scheduling`: reusa `GET/POST /store/slots` (manager/admin); adicionado `DELETE /store/slots/:id`.
- Testes: `apps/api/src/admin/admin-merchants.service.spec.ts` (10 casos — list/create/CRUD/lock/staff).
- Arquivos: `apps/api/src/admin/admin-merchants.{service,controller,service.spec}.ts` (+ registro em `admin.module.ts`); `apps/api/src/scheduling/scheduling.{service,controller}.ts`; `apps/admin/src/pages/merchants/{MerchantsList,MerchantDetail,StoreDetail}.tsx` (+ `App.tsx`, `components/Layout.tsx`, `styles.css`).
